# 受注提案用サマリー

## 1. 提案コンセプト

「常連管理」「LINE再来店促進」「来店履歴蓄積」を一つの画面で回せる、スナック店舗専用のCRMを提供する。

一般的な飲食店向け予約台帳ではなく、以下を前提にした業務特化型にする。

- 顧客ごとの会話ネタ、同伴傾向、好み、ボトル情報を蓄積できる
- ママ、店長、キャストそれぞれで使い分けられる
- LINEの一斉配信と個別フォローを分けて管理できる
- 来店頻度の低下を自動検知して再来店アクションにつなげられる

## 2. この案件で刺さる価値

- 紙台帳やExcel管理から脱却できる
- 属人的な顧客管理を店舗資産に変えられる
- 「誰にいつ連絡したか」が残る
- 来店前後の接客メモが残るため新人教育にも効く
- 休眠顧客の掘り起こしを仕組み化できる

## 3. 主機能

### 顧客管理

- 顧客基本情報
- ニックネーム、年代、居住エリア、勤務先メモ
- 好きな酒、カラオケ、会話テーマ、NG事項
- 担当キャスト、同伴者、紹介元
- ランク管理
- 休眠判定

### LINE配信管理

- 友だち連携
- セグメント配信
- 一斉配信
- 個別送信履歴
- テンプレート管理
- イベント告知配信
- 配信禁止管理

### 来店履歴管理

- 来店日時
- 同伴有無
- 同席キャスト
- 滞在時間
- 飲食内容
- 会計金額
- ボトル追加
- 接客メモ

## 4. 受注時の推奨パッケージ

### Phase 1: 最小導入版

- 顧客台帳
- 来店履歴
- LINE個別メモ
- セグメント配信
- ダッシュボード

### Phase 2: 運用強化版

- 自動休眠検知
- テンプレート配信
- ボトル期限通知
- 担当キャスト別集計
- イベント集客機能

### Phase 3: 高度化版

- LINE webhook連携による会話ログ取り込み
- LTV分析
- 来店予測
- レコメンド配信
- 複数店舗対応

## 5. 提案時に押さえるべき論点

- LINE公式アカウントの運用主体は店舗か、運営会社か
- 1店舗運用か、多店舗展開前提か
- 個人情報の取り扱い責任者
- キャストが私物スマホから触る前提か
- レシート連携や会計システム連携の必要有無
- LINEの個別チャットをどこまでシステム上で扱うか

## 6. 推奨技術構成

- フロント: Next.js
- API: Next.js Route Handlers または NestJS
- DB: PostgreSQL
- 認証: Clerk または NextAuth
- バックエンド非同期処理: Supabase Queue 相当または cron + job runner
- ファイル保管: S3 互換ストレージ
- LINE連携: Messaging API + LIFF
- インフラ: Vercel + Supabase または AWS

## 7. 受注提案での差別化ポイント

- スナック業態に合わせた業務用語で画面設計している
- 一般的なメルマガ機能ではなくLINE運用前提
- 「来店履歴」と「配信結果」が同一顧客軸で見える
- 初期導入しやすいMVPと、将来拡張の両方を設計済み

## 8. ざっくり見積もりの切り方

- 要件定義: 1.5週間
- UI/UX設計: 1週間
- MVP開発: 5から7週間
- テストと導入支援: 1.5週間

合計で約2か月から2.5か月を基本線とする。
